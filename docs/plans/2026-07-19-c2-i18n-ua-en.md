# План C2: локалізація UA/EN (для виконавця на молодшій моделі)

Давно відкладений план (відкладався з Етапу C). Мета: весь видимий UI
двомовний — українська за замовчуванням, англійська через перемикач.
Бета буде з українськими астрологами, тож укр-локаль — не прикраса,
а умова беты.

**Порядок відносно решти ROADMAP:** цей план виконується ПЕРЕД повним
security-рев'ю. Причина: локалізація торкнеться майже кожного
фронтенд-файлу, і рев'ю коду, який от-от буде перелопачений, знецінюється.
Рев'ю — наступний етап після C2, по фінальному коду.

## Ключові архітектурні рішення (прийняті, не переглядати без згоди власника)

1. **БЕЗ URL-роутингу локалей.** Жодних `app/[locale]/…`, жодного переносу
   файлів. next-intl працює в режимі "without i18n routing": локаль
   визначається cookie `NEXT_LOCALE`, дефолт — `uk`. Причини: (а) наш
   `proxy.ts` — власний auth-middleware зі списками PUBLIC_PATHS по
   `startsWith` — композиція з i18n-middleware то окреме мінне поле;
   (б) перенос 39 файлів у `[locale]` — величезний churn без потреби.
   Ціна: немає окремих URL для мов (SEO/hreflang) — свідомо відкладено
   пост-launch, зафіксовано в «Не робити».
2. **Дефолт — українська.** Ринок український, `html lang="uk"` вже стоїть,
   легальні сторінки українською. Новий відвідувач без cookie бачить укр.
3. **Легальні сторінки (/privacy, /terms) НЕ перекладаються** — лишаються
   українською в обох локалях. Юридичний текст в одному екземплярі, без
   ризику розбіжностей перекладу. (На EN-локалі вони просто українські.)
4. **Бекенд не чіпаємо**, окрім двомовних email-шаблонів (Частина 5).
   Фічі тарифів приходять з бекенда англійською (PLANS у billing.py) —
   фронтенд перекладає їх словником «рядок → переклад» з fallback на
   оригінал. API-помилки (detail-рядки) лишаються англійською.

## Частина 1 — Інфраструктура

### 1.1. Установка й конфіг
- `npm i next-intl` (4.13.x).
- Прочитати в доках next-intl розділ **"App Router → without i18n routing"**
  (доки в node_modules або на сайті) — сетап відрізняється від роутованого!
- `i18n/request.ts`: `getRequestConfig` — локаль з cookie `NEXT_LOCALE`
  (через `cookies()` з `next/headers`), валідні значення `uk | en`,
  усе інше/відсутнє → `uk`.
- `next.config.ts`: обгорнути `createNextIntlPlugin()` — існуючий
  `headers()` зберегти. Після: `curl -sI http://localhost:3000/login` —
  всі 5 заголовків на місці.
- `messages/uk.json`, `messages/en.json` — скелет неймспейсів:
  `common`, `nav`, `auth`, `dashboard`, `persons`, `charts`, `astro`,
  `billing`, `pricing`, `account`, `landing`, `errors`.
- `app/layout.tsx`: `NextIntlClientProvider` + `html lang={locale}`
  (зараз hardcoded "uk"); metadata поки не чіпати (Частина 4).

### 1.2. Перемикач мови
- Server action `setLocale(locale)`: ставить cookie `NEXT_LOCALE`
  (httpOnly НЕ потрібен — це не секрет; maxAge ~рік, path=/) і робить
  `revalidatePath("/", "layout")` або еквівалент, щоб серверні компоненти
  перерендерились.
- Клієнтський `LanguageSwitcher` (компактний «UK | EN», активна мова
  виділена): розмістити (а) у футері сайдбара дашборда поруч із
  Privacy/Terms, (б) у мобільному хедері або мобільному NAV-рядку,
  (в) на (auth)-layout і лендингу (кут сторінки).
- Верифікація: перемкнути → УСЯ сторінка (включно з server components)
  перерендерилась іншою мовою; cookie видно в devtools; після
  повного перезавантаження мова збереглась.

### 1.3. Перевірка виживання куки при токен-рефреші
Залогінитись, дочекатись протухання access_token (30 хв — АБО тимчасово
виставити менший `ACCESS_TOKEN_EXPIRE_MINUTES` в .env бекенда для тесту),
зробити навігацію, що тригерить рефреш у proxy.ts → локаль НЕ скинулась.
Якщо скинулась — фікс у proxy.ts (зшивання Cookie-хедера губить куки).

## Ризики
1. next-intl-плагін у next.config.ts може загубити `headers()` — перевірити
   curl-ом одразу в Частині 1, не в кінці.
2. `proxy.ts` вручну зшиває Cookie-хедер при рефреші — NEXT_LOCALE може
   загубитись саме там; тест у 1.3 обов'язковий.
3. Патерни v4 next-intl (getTranslations/useTranslations/setRequestLocale)
   можуть відрізнятись від training data — читати доки пакета, не пам'ять.
4. Укр плюрали (one/few/many/other) — не забути `few` (2-4 карти).
5. Не перекласти повідомлення-стан (помилки форм) — легко пропустити;
   хардкод-аудит у Частині 6 їх ловить.
6. Подвійна поверхня друкарських помилок — фінальна вичитка укр текстів
   одним проходом.
7. Великий diff по 30+ файлах — комітити по частинах, не одним мегакомітом.

## Не робити (поза скоупом)
- URL-роутинг локалей (`/en/...`), hreflang, локалізований sitemap —
  окрема пост-launch SEO-задача.
- Переклад privacy/terms (лишаються українською в обох локалях).
- Локалізація API-помилок бекенда і збереження локалі юзера в БД.
- Переклад назв Arabic parts (100+ рядків з бекенда) і назв міст атласу.
- Треті мови, RTL.
