# Zorya — план розвитку та виходу в продакшен

> Оновлено: 2026-07-15
> Стан на момент написання: робочий прототип — auth із refresh-flow, рушій на Swiss Ephemeris
> (натал / транзити / соляр / синастрія), атлас на 34k міст, арабські частини, терми,
> SVG-колесо з тултіпами й градусами куспідів, модуль персон, денний ліміт free-плану,
> Stripe/LiqPay підключені, але з відомими багами. 32 тести проходять.
> Історичний аналіз ZET9 і початковий план — див. [PLAN.md](PLAN.md).

---

## ⚠️ Блокери, які треба вирішити ДО прийому грошей

### 1. Ліцензія Swiss Ephemeris
`pyswisseph` — AGPL. Для закритого комерційного SaaS це означає:
- **або** відкрити код бекенду (AGPL-зобов'язання),
- **або** купити комерційну ліцензію Astrodienst (одноразово ~CHF 750).

Рішення потрібне до публічного запуску.

### 2. Обіцянки Pricing-сторінки
Pro-план зараз обіцяє **PDF export, custom orbs, progressions** — жодного немає в коді.
До першого платежу: реалізувати або прибрати з опису тарифу.

### 3. Назва
«Zorya» обрана як робоча, але домен і торгова марка **не перевірені**.
Перевірити zorya.app / zorya.com.ua та збіги в App Store перед брендингом.

## Етап A — Стабілізація (тиждень 1): закрити відоме зламане

- [ ] **CI**: GitHub Actions — pytest + `tsc --noEmit` + `next build` на кожен push/PR
- [ ] **Білінг** (знахідки рев'ю 2026-07-13):
  - [ ] Stripe: metadata через `subscription_data` + обробка `checkout.session.completed`;
        зберігати `stripe_customer_id` на користувачі
  - [ ] Фікс `stripe.errors.*` → `stripe.error.*` (зараз webhook падає 500 на невалідному підписі;
        `ValueError` від битого payload теж не ловиться)
  - [ ] Підключити `require_plan` до transit / solar-return / synastry
        (зараз Pro-фічі доступні free-юзерам у межах денного ліміту)
  - [ ] LiqPay: обробка `unsubscribed` / `failure` (зараз скасована підписка = вічний Pro)
  - [ ] `API_PUBLIC_URL` у settings замість хака `frontend_url.replace('3000','8000')`
  - [ ] Писати в таблиці `subscriptions` / `payments` (зараз порожні — немає аудиту грошей)
- [ ] **Безпека auth**: rate limiting (slowapi) на login/register; ротація refresh-токенів
- [ ] **Точність**: SE1-файли (з ZET9 `Swiss/` або Astrodienst) → `EPHE_PATH`
      (вмикає Хірон і максимальну точність); 3–5 golden-тестів куспідів проти astro.com

## Етап B — Інфраструктура (тижні 2–3): продакшен-середовище

- [ ] **PostgreSQL** замість SQLite: asyncpg, alembic-міграція вже готова;
      пошук міст — `pg_trgm` замість SQLite FTS5
- [ ] **Хостинг** (рекомендація):
  - фронтенд → Vercel (безкоштовний tier, SSR з коробки)
  - бекенд + Postgres → Docker Compose на Hetzner VPS (~€5/міс) за Caddy (авто-HTTPS)
  - LiqPay server-callback працює на VPS без сюрпризів
- [ ] **Домен**: перевірити та купити (див. блокер №3)
- [ ] **Операційка**:
  - [ ] Sentry (frontend + backend)
  - [ ] Uptime-моніторинг (UptimeRobot / Betterstack)
  - [ ] Щоденний `pg_dump` у S3/Backblaze + тестове відновлення
  - [ ] Секрети через env; fail-fast на дефолтний `SECRET_KEY` уже реалізовано

## Етап C — Продуктова готовність (тижні 3–4)

- [ ] **Email-флоу**: підтвердження пошти + відновлення пароля (Resend) —
      `forgot-password` був у плані Фази 1, але не реалізований
- [ ] **Збереження карт**: таблиця `charts` (person_id, type, config, data JSONB) —
      зараз розрахунки зникають після перезавантаження; найдешевша фіча
      з найбільшим утриманням користувача
- [ ] **Landing page** (зараз `/` просто редіректить на /login) + SEO-мета
- [ ] **Локалізація UA/EN** (next-intl) — для українського ринку з LiqPay це must
- [ ] **Юридичне**: Privacy Policy, Terms, `DELETE /users/me` (GDPR),
      політика зберігання даних народження
- [ ] **Мобільна адаптація**: колесо та форми на 375px

## Етап D — Запуск (тиждень 5)

- [ ] Security-рев'ю всієї гілки + виправлення знахідок
- [ ] E2E-тест оплати: Stripe test mode і LiqPay sandbox
      (оплата → webhook → план → скасування → downgrade)
- [ ] Закрита бета на 10–20 користувачів — насамперед астрологи-практики:
      вони одразу зловлять розбіжності точності, а точність — головна цінність продукту
- [ ] Plausible analytics (GDPR-friendly, без cookie-банера)
- [ ] Публічний запуск
